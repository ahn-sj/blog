이 글의 질문은 하나다. Redis가 죽고 다시 켜졌을 때 무엇을 얼마나 잃어도 되는가?

RDB와 AOF는 단순한 백업 기능이 아니다. Redis에 들어간 데이터가 장애 뒤 어디까지 남아야 하는지 정하는 선택이다.

![Redis RDB와 AOF 비교](/images/redis-persistence-rdb-aof.svg)

## 먼저 데이터 성격을 나눈다

Redis에 들어가는 값이 모두 같은 무게를 갖지는 않는다.

| 데이터 | 사라졌을 때 영향 | 복구 방법 | 손실 허용 |
| --- | --- | --- | --- |
| Cache | 원본 DB를 다시 읽는다 | 다시 채운다 | 높은 편 |
| Session | 로그인이 풀린다 | 사용자가 다시 로그인한다 | 낮은 편 |
| Rate limit counter | 제한이 잠시 풀릴 수 있다 | 시간이 지나 다시 쌓인다 | 정책에 따라 다름 |
| Ranking score | 순위가 틀릴 수 있다 | 원천 데이터로 재계산한다 | 재계산 가능 여부에 따라 다름 |

세션 데이터가 사라지면 사용자는 다시 로그인해야 한다. 이 한 문장이 RDB와 AOF 선택을 가른다. “조금 잃어도 되는가?”에 답하지 못하면 설정을 골라도 운영 설명이 되지 않는다.

## RDB는 시점 저장이다

RDB는 특정 시점의 Redis 상태를 파일로 남긴다. 사진을 찍는 방식에 가깝다.

| 장점 | 비용 |
| --- | --- |
| 파일이 비교적 작다 | 마지막 저장 뒤 변경분은 사라질 수 있다 |
| 재시작이 빠른 편이다 | 저장 주기와 손실 범위를 함께 설명해야 한다 |
| cache처럼 다시 만들 수 있는 데이터에 잘 맞는다 | 최근 쓰기가 중요한 데이터에는 불안할 수 있다 |

cache 위주 Redis라면 RDB만으로도 충분할 수 있다. 핵심은 장애 뒤 cache miss가 늘어나는 정도를 받아들일 수 있는지다.

## AOF는 쓰기 기록이다

AOF는 Redis에 들어온 쓰기 명령을 기록한다. 사진보다 일지에 가깝다.

| 장점 | 비용 |
| --- | --- |
| 최근 데이터 손실을 줄이기 좋다 | 파일이 커질 수 있다 |
| 세션처럼 최근 값이 중요한 데이터에 맞기 쉽다 | rewrite 중 지연을 관찰해야 한다 |
| 장애 뒤 설명 가능한 손실 범위를 만들기 좋다 | 디스크 비용과 운영 복잡도가 늘어난다 |

AOF를 켠다고 손실이 완전히 사라지는 것은 아니다. 하지만 “어느 정도까지 잃을 수 있는가”를 더 좁게 설명할 수 있다.

## 선택은 이렇게 기록한다

| 질문 | RDB 쪽 답 | AOF 쪽 답 |
| --- | --- | --- |
| 마지막 몇 초, 몇 분을 잃어도 되는가 | 가능할 때 선택하기 쉽다 | 어려울 때 선택하기 쉽다 |
| 재생성할 원본이 있는가 | 있으면 부담이 낮다 | 없어도 손실을 줄일 수 있다 |
| 장애 뒤 사용자 영향이 큰가 | 작을 때 맞다 | 클 때 맞다 |
| 디스크와 지연 비용을 감당할 수 있는가 | 비교적 낮다 | 더 봐야 한다 |

좋은 Redis 설정은 멋진 설정이 아니다. 장애 뒤 어떤 값이 사라질 수 있는지 설명할 수 있는 설정이다.
